RAG Pipelines with Vector Embeddings

08.12.2024
12 min read
Harsh Parashar

Large Language Models hallucinate. They don't know about your private data. Retrieval-Augmented Generation (RAG) solves this by injecting relevant context directly into the prompt window.

01. The Context Window Problem

Even with GPT-4's 128k token window, you cannot fit an entire corporate knowledge base into a single prompt. It's expensive and slow. We need a way to search for semantically relevant snippets of text and only send those to the LLM.

02. Creating Embeddings

We use models like OpenAI's `text-embedding-3-small` to convert text chunks into high-dimensional vectors (arrays of floats). 

import openai
from pinecone import Pinecone

# 1. Generate Embedding
def get_embedding(text):
    response = openai.embeddings.create(
        input=text,
        model="text-embedding-3-small"
    )
    return response.data[0].embedding

# 2. Vector: [-0.002, 0.054, -0.012, ... 1536 dims]
vector = get_embedding("How do I reset my password?")

03. The Retrieval Loop

Once data is indexed in a Vector DB (like Pinecone or Milvus), the runtime flow looks like this:

  1. User asks a question.
  2. Convert question to Vector query.
  3. Perform Cosine Similarity search in DB.
  4. Retrieve top 3 matching chunks.
  5. Construct prompt: "Answer using this context: {chunks}".

04. Summary

RAG bridges the gap between static LLM knowledge and dynamic, private data. It is the cornerstone of building "Chat with your PDF" apps or enterprise support bots.
