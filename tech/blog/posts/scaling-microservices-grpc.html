Scaling Microservices: The gRPC Protocol

14.12.2024
8 min read
Harsh Parashar

As microservice architectures scale beyond a handful of services, the chattiness of HTTP/1.1 REST APIs often becomes a bottleneck. Here's why we migrated our core communication layer to gRPC.

01. The REST Bottleneck

JSON is human-readable, which is great for debugging but terrible for performance. In a high-throughput system processing millions of inter-service requests, the overhead of text-based serialization (parsing strings into objects) adds up significantly.

Furthermore, REST often suffers from over-fetching or under-fetching data, and it lacks strict schema enforcement, leading to "integration drift" where services break because a field name changed silently.

02. Protocol Buffers

At the heart of gRPC lies Protocol Buffers (protobufs). Unlike JSON, protobufs are a binary serialization format. You define your schema once in a .proto file, and code is generated for any language (Go, Python, Java, Node, etc.). 

// service.proto definition
syntax = "proto3";

package order;

service OrderService {
  // Unary RPC
  rpc CreateOrder (OrderRequest) returns (OrderResponse);
  
  // Server Streaming RPC
  rpc StreamOrders (StreamRequest) returns (stream OrderResponse);
}

message OrderRequest {
  string user_id = 1;
  repeated string item_ids = 2;
  double total_amount = 3;
}

The numeric tags (= 1, = 2) are crucial. They allow fields to be renamed without breaking backward compatibility, as long as the tag number remains consistent.

03. HTTP/2 & Multiplexing

gRPC is designed on top of HTTP/2. This provides several massive advantages over HTTP/1.1:

  • Multiplexing: Multiple requests can be sent over a single TCP connection in parallel. No more head-of-line blocking.
  • Bi-directional Streaming: Client and server can stream data to each other simultaneously.
  • Header Compression: HPACK compression reduces overhead significantly.

04. Conclusion

Migrating to gRPC isn't free. It adds complexity to the build process (managing .proto files) and makes ad-hoc curl testing harder (though tools like grpcurl exist). However, for internal microservice communication, the type safety and performance gains are undeniable.
